# Changelog

All notable changes to TeXLib are recorded here. The format follows [Keep a Changelog](https://keepachangelog.com/en/1.1.0/); versions correspond to git tags.

## [Unreleased]

### Changed

- **The native Sublime plugin gains a live status bar, editor delegations, domain depth, and a Doctor — and no longer installs classes into TEXMF (it *uninstalls* a shadowing copy instead).** Builds show a status-bar spinner + current step (parallel-safe across concurrent document builds), report success-with-warnings, and color the panel theme-agnostically (`region.redish`/`region.yellowish`). New commands: Word Count / Table of Contents / Jump-to-`\ref` (delegated to LaTeXTools), Edit Settings, Insert Problem by Topic, Bank Report, Bank Preview (via `bank.cls`), Show Resolved Metadata, Toggle Build on Save, and **`TeXLib: Doctor`** (checks the toolchain, `texinputs`, coursemeta resolution, and whether a `TEXMFHOME` copy shadows your checkout). Coursemeta keys complete inside `\metasetup{ }`. The former "Install Classes to TEXMF" is replaced by **Uninstall Classes from TEXMF**: the plugin's builds already resolve the classes from your checkout via `texinputs`, so a `TEXMFHOME` copy only *shadows* it (kpathsea/LUAINPUTS can find the stale copy first); the command removes it, and a one-time build-time warning nudges you when a shadow is present. System-wide install stays the standalone TeXLib-Installer's job; the plugin's own distribution channel is decided as a Package Control custom repository (`messages.json` scaffolded).
- **The boxed-theorem taxonomy is unified behind one call + short aliases.** `\TeXLibAllTheorems[options]` declares the whole catalog (the full list now lives once in `texlib-thmenv.sty`), and every environment gains a short alias (`\thm \cor \lem \conj \prop \proc \rmk \rcl \nte`, alongside the existing `\defn \conv …`). `didactic` and `pset` call `\TeXLibAllTheorems` (pset gains `exercise`/`question` + the aliases; didactic gains `note`).
- **The solution box's appearance is sourced from one place.** `texlib-corepkg.sty` now defines `\texlibsoltint` / `\texlibsolrulecolor` / `\texlibsolblankrulecolor` / `\texlibsolheader`, referenced by both the exam/quiz shared box (`texlib-solutions.sty`) and pset's own box, so the solution look retunes once. The two keep different box *mechanisms* on purpose — `\colorbox`/`\box` for working SyncTeX reverse search vs pset's tcolorbox for the `\StudentMode` watermarked blank-answer box.
- **`Sublime/texlib_builder.py` is now a thin LaTeXTools adapter over the plugin's shared build core (1690 → 113 lines).** The build logic that used to live in the LaTeXTools builder was extracted into the host-agnostic `TexlibBuildCore` (`Sublime/texlib/texlib_build.py`), which both the native plugin (`TexlibBuild`, supplying the host contract via a constructor) and the LaTeXTools builder (`TexlibBuilder(TexlibBuildCore, PdfBuilder)` — core first in the bases so `commands()`/`_*` resolve to it, `PdfBuilder` supplying `__init__` and the host attributes: `display` / `tex_root` / `engine` / `options` / `out` / `aux_directory`) now share. One source of build behavior, nothing to drift. `Ctrl+B` runs the native path; `Tools → Build With → TeXLib` still runs the adapter. The console-flash `Popen` monkeypatch stays in the adapter (LaTeXTools spawns those build processes; the native runner owns its own `Popen` and sets `CREATE_NO_WINDOW` directly).
- **`\ShowKey` / source `\keys` now builds the exam answer key.** Previously `\ifkey` was never consulted in `autoexam.cls`, so `\def\ShowKey{}` was a silent no-op and the key had to be requested via `\ShowSolutions`. `autoexam.cls` now maps `\ifkey` to the instructor-only build (`AutoExamSolMode=only`) and forces `\ifsolutions`, so `\ifkey` implies `\ifsolutions` — an exam's answer key IS its instructor copy, and `\ShowKey`/source `\keys` request it directly. `test_mode_effects.py`'s ShowKey row pins this (asserts `\ShowKey` reveals identically to `\ShowSolutions` for a no-`\versions` fixture).
- **The autoexam/quiz question shuffle is now a typeset-time list permutation, not a source-text rewrite.** `\shuffle` used to run a pre-pass that re-read the document off disk, regex-parsed the `\begin{problems}` blocks with a hand-rolled partial-TeX tokenizer, Fisher-Yates-permuted the string fragments, and wrote a per-version temp `.tex` to `\input`. `{problems}`/`{mcproblems}` now put `\problem` into *collect* mode — each item records its args instead of typesetting, `\newpage` marks a page boundary, `\section` marks a hard shuffle boundary, `\extracredit` defers to the end — and `\end` emits the section permuted per version through the existing `pbank_problem_item` path (`pbank_emit_section` in `problem_engine.lua` + a new pure `problem_shuffle.lua`, unit-tested under `texlua` via `test_shuffle.lua`). The authoring syntax is unchanged and existing exams need no edits; a non-shuffled exam is byte-identical to before. Because draws now happen at emit time, a `\shuffle` exam's per-version orderings differ from the old scheme — a one-time reshuffle (deterministic and reproducible as before). This deletes ~170 lines of fragile string-parsing (`shuffle_problems_body` et al.), collapses the per-version body temps to one, and removes the whole class of `\newpage`-as-delimiter and misparse fragilities. The RNG is a per-version-seeded MINSTD stream with high-bit extraction + a SplitMix32 avalanche, so even small exams give every version a distinct, uniform order. Known gap: the `.sco`/`\scorepage` instructor grid still reflects authored order (only wrong under `\shuffle` + `\scorepage` + per-problem-varying points at once; uniform-point sections are correct). Also: `problem_engine.lua`/`texlib_synctex.lua` scratch now routes to a hashed system-temp subdir when neither `TEXLIB_AUX_DIR` nor `-output-directory` is set, so even a bare `lualatex doc.tex` leaves zero engine scratch (body/`.sco`/`.srcmap`/synctex) beside the source.
- **Weekly quiz-day auto-quizzes are suppressed on exam days.** When a `quiz-days` weekday lands on the same day as an `\exam` (common on a compressed summer calendar, where an exam placed on a Thursday collides with a Thursday quiz day), the schedule engine no longer stacks an automatic "Quiz N" on top of the exam — an exam and a quiz on the same day is virtually never intended. `render_grid`'s auto-quiz pass in `schedule.lua` now also skips a cell flagged as an exam day. The two explicit opt-outs are unchanged and still available for other cases: `\noquiz{<date>}` (suppress one day) and the `\exam[noquiz]` option (suppress the whole exam week). The suppression applies **only** to the automatically-generated quizzes — an explicit `\quiz[date=<exam day>]` still renders on the exam day (it is appended directly, independent of the auto pass), so a deliberately co-scheduled quiz is preserved. Existing schedules that relied on a stray exam-day auto-quiz will lose it and renumber; courses that worked around this with a hand-placed `\noquiz{<exam date>}` can drop it (still harmless if kept). Covered by `Schedule/test_schedule_quiz_exam.lua`.
- **Exam and quiz instructions are auto-wrapped in the titled "Instructions:" box — including instructions *files*.** Previously only inline/default instructions were boxed; a user-set `exam-instructions-file`/`quiz-instructions-file` rendered *unboxed* ("controls its own layout"), which forced every course instructions file to hand-roll the `\fbox{\fbox{\parbox{\dimexpr\linewidth-4\fboxsep-4\fboxrule\relax}{\large\bf Instructions:}…}}}` boilerplate itself. `autoexam.cls` and `quiz.cls` now render all three paths (file/inline/default) through a single shared `\TeXLibInstructionsTitled{<prefix>}` renderer in `texlib-instructions.sty`, so instructions files can be just their content. Quizzes gain the bold "Instructions:" header and full-width box they lacked (they previously used the centered, `\footnotesize`, fixed-6.25in `\TeXLibInstructionsBox`, which is unchanged and still used by `bingo`). A document that genuinely needs full layout control over its own instructions file can opt out with `\TeXLibInstructionsUnboxed` in its preamble (bare, no box). Exam-cover output for existing inline/default exams is byte-identical; **file-based** exam/quiz instructions now pick up the box, so any course file that hand-rolled the box must drop it to avoid a double box (Math 182's `exam-instructions.tex`, `exam-instructions-series.tex`, and `quiz-instructions.tex` were updated accordingly).

### Added

- **`bank.cls` — a first-class problem-bank catalog/preview class.** A thin `\documentclass{bank}` wrapper `\loadbank`s a bare-fragment bank and `\printbankcatalog`s it — every problem with its id, `[attrs]`, stem, and (always-shown) solution, in a clean catalog page style — using the same problem/solution stack the assessment classes use. It replaces the Sublime builder's synthesized-quiz preview harness and backs `TeXLib: Bank Preview`. (`Bank/bank.cls`, `Bank/bank-template.tex`.)
- **`<jobname>.metadump` sidecar + `TeXLib: Show Resolved Metadata`.** `course-metadata.sty` dumps every resolved field at `\AtEndDocument` (coursemeta defaults + class-option overrides + derived fields like `Term`); the plugin reads it to show the fully-resolved metadata for a document. Written every build, routed to the aux dir like `.pubmeta`, gitignored.
- **`bank_report.py` + `collate_keys.py` — standalone bank/exam tooling.** `bank_report.py` prints a bank coverage dashboard (topic × difficulty matrix, duplicate/unused/dangling ids, and an exam-length estimate). `collate_keys.py` merges per-version `<base>_<V>_solutions.pdf` slices into one bookmarked `<base>_AllKeys.pdf` (pypdf).
- **Render-correctness test suites for the visual output** — real builds that assert *rendering*, not just content presence: `test_bank_catalog.py` (catalog part labels render `(a)/(b)`, solutions shown), `test_class_render.py` (theorem labels + short aliases resolve + the section-based shared counter numbers `1.1…1.5`; the C2 solution header), and `test_mc_schedule_render.py` (the MC key marks the *correct* choice — `Answer: C` for a `\cchoice` in slot 3, not `A`; the schedule grid renders headers / holiday / `No Class` / Quiz / Exam). Wired into the `tests.yml` real-toolchain job.
- **CI: an advisory `chktex` lint job on every branch, and a `workflow_dispatch` module-subset input for the (main-only) smoke build.**
- **Native Sublime Text plugin (`Sublime/texlib/`) — TeXLib owns the build; LaTeXTools becomes a companion it *calls*.** A self-contained Sublime package that is the primary build path (`Ctrl+B`) without depending on LaTeXTools' internals. The build logic is a host-agnostic core (`TexlibBuildCore` in `texlib_build.py`) driven by a native async runner (`texlib.py`: `Popen`-per-engine-pass, streamed output panel, cancel/kill, the cross-reference rerun loop, biber change-detection cache, aux-directory routing, autoexam per-version PDF slicing, and the publish step) — everything the LaTeXTools builder did, through the plugin's own process management instead of `PdfBuilder`. Editor smarts it deliberately does *not* rebuild are delegated to LaTeXTools by stable `run_command` name (post-build PDF open + forward-sync via `latextools_jumpto_pdf`; `View PDF` / `Forward Sync` commands), never by importing its internals — coupling is a documented command name, not a fragile base class. It adds TeXLib-domain commands LaTeXTools has no concept of: bank navigation (go-to / insert `\getproblem{id}`, plus problem-**id** completions inside `\getproblem{…}`), new-document scaffolding from `<class>-template.tex`, `Open coursemeta.tex` / `Reveal` + `Clean Aux Directory` locators, `Package for LMS`, and `Install Classes to TEXMF` (copies the `.cls/.sty/.lua` payload into `TEXMFHOME/tex/latex/texlib/` so *every* TeX tool — CLI, other editors, CI — resolves the classes, i.e. the installer's job done for one machine, without editor-capturing the classes). Plus macro/environment completions and snippets, a `Tools → TeXLib` menu, and machine-local `TeXLib.sublime-settings` (`texinputs`, `open_pdf_on_build`, publish toggles, `class_source`). Each domain command is its own top-level file so it hot-reloads independently on save; every piece has a headless test suite (`Sublime/test_texlib_*.py`, no TeX toolchain needed). Design rationale + phase history in `Sublime/PLUGIN-DESIGN.md`.
- **Publish step: syllabus / schedule builds emit shareable copies + a desktop shortcut.** A class now opts in with `\TeXLibDeclarePublishable{<kind>}{<generic>}{<noun>}` (only `syllabus` and `schedule` do); `course-metadata.sty` then drops a `<base>.pubmeta` sidecar at `\AtEndDocument` recording the *resolved* course / section / term (coursemeta defaults + class-option overrides + the derived `\GetTerm`, none of which a build tool can reconstruct by scraping `coursemeta.tex`). On a normal Sublime build the builder's `_postprocess` reads it and, next to the source, clones the PDF to `<course>.<section>_<term>.pdf` (e.g. `Math181.1001_Fall2026.pdf`, section segment dropped when unset) and the generic `<generic>.pdf` (`Syllabus.pdf` / `Tentative Schedule.pdf`), drops a `"<course> <term> <noun>"` shortcut in the desktop's `Course Materials` folder (`WScript.Shell`, `GetFolderPath('Desktop')` so the OneDrive-redirected desktop resolves), and copies the shareable path to the clipboard for pasting into the LMS. An optional `publish-name` coursemeta key overrides the coded basename. The mechanism is metadata-driven — any future class opts in with the one declaration line, no builder change. Enabled by default; toggle via `builder_settings` `publish_shareable_copies` / `copy_published_path_to_clipboard` (or env `TEXLIB_PUBLISH` / `TEXLIB_PUBLISH_CLIPBOARD`). **Sublime-only** — deliberately NOT mirrored into the installer's `texlib-build.ps1` (coworkers don't want auto desktop shortcuts). The same-file copy guard is case-insensitive (`normcase` + `samefile`): a source literally named `syllabus.tex` and the generic `Syllabus.pdf` are one file on a case-insensitive volume, so the generic copy is skipped (the source already is it) rather than deleting the just-built PDF — the coded copy + shortcut are still made. A CLI build only *writes* the sidecar (no builder = no copies); it's gitignored.
- **`package_for_lms.py` — bundle a course's shareable PDFs for LMS upload.** A self-contained standalone script: walks up from a dir/file to the course root (the folder with `coursemeta.tex`), collects the published `Syllabus.pdf` + `Tentative Schedule.pdf`, and zips them into `<Course>_<Term>_LMS.zip` (name read from `coursemeta.tex`). `--list` previews, `-o` overrides the output path.
- **Build-summary line.** Every build now ends with a one-liner: elapsed time, engine passes, biber runs, and the output PDF's size (`_count_passes` tallies the stream; `_display_build_summary` reports it).
- **`Sublime/test_synctex_integration.py` — real end-to-end inverse-search test.** Drives the actual Sublime builder against a real `lualatex` build, then uses TeX Live's own `synctex edit -o page:x:y:pdf` CLI to simulate a SumatraPDF double-click and check where it actually lands. The existing SyncTeX tests (`Schedule/test_schedule_synctex.lua`, `test_texlib_builder.py`'s schedmap-rewrite cases) both fabricate their input data by hand and can't catch a real engine/table-package interaction — which is exactly what this found (see Fixed, below). Wired into CI (`tests.yml`, `biber-integration` job; needs poppler's `pdftotext` for `-bbox`, added via `apk add poppler-utils`).
- **`test_engine_correctness.py` + `test_exam_seed.lua` — rendered coverage for the problem engine's "wrong number / wrong answer key" silent-failure class.** `tests/fixtures/Exams/fix-test.tex` only proves the `\problem{id}[a=1,...]` override *syntax* parses, and `\calcvar` had no buildable fixture at all — so a regression that substituted the wrong value or keyed the wrong multiple-choice letter would have shipped silently. `test_engine_correctness.py` drives a real `lualatex` build of committed bank fixtures and reads the PDF back with poppler's `pdftotext`, asserting the *exact* rendered values: fixed-variable overrides actually beat the bank's own `\setvar`/`\setrng` (`fixed[]` guards), `\calcvar` integer and sqrt-style results, the MC answer letter matches whatever slot the `\cchoice` option rendered in (deterministic and under `\shuffle`), `[fixed]` order preservation, `\fchoice[i]` slot pinning, `[choose=m]` selection, student/key option-order parity, `\ppart` part atomicity/sub-numbering, `\importproblem` (standalone-file render + override lock + `push_scope`/`pop_scope` isolation), and the engine's `.vmap` version-marker emission. `test_exam_seed.lua` is a pure-`texlua` unit test of `set_exam_seed`'s per-version decorrelation (distinct seeds *and* distinct problem-order permutations for small item counts — the historically collapse-prone path). Both are wired into CI (`tests.yml`, `biber-integration` job); the Python suite soft-skips cleanly without `lualatex`/poppler, matching the sibling integration tests.
- **`test_mode_effects.py` — assessment build-flag leakage guard, wired into every-push CI.** The shuffle permutation core (`test_shuffle.lua`) was invoked by no workflow — `smoke.yml`/`visual.yml` are main-only, so on a feature branch the headline permutation core was ungated. It now runs (`texlua test_shuffle.lua`) in `tests.yml`'s `biber-integration` job on every push, so a broken permutation fails CI immediately. Alongside it, the new `test_mode_effects.py` closes a second hole: `smoke_test.py` builds each module under every mode but checks `EXPECT_TEXT` by module only, so `Exams --default` and `Exams --solutions` were asserted against the *identical* token list and nothing verified the six `texlib-build.sty` flags actually gate content — a flag silently becoming a no-op, or a solution/answer/rubric leaking into a student copy (a FERPA-adjacent failure), passed CI. `test_mode_effects.py` builds one self-contained `autoexam` fixture (one free-response `{solution}`, one multiple-choice problem with `\rubric`) once per compile-time flag (`\def\ShowSolutions{}`/`\def\ShowRubric{}`/…, the builder's own injection path) and asserts a differential, mode-keyed map: the solution token, the multiple-choice answer-letter badge, and the `\rubric` overlay are ABSENT from every student build and PRESENT only under the flag that reveals them. It soft-skips (like the biber/synctex tests) when `lualatex` or a poppler `pdftotext` is missing. Its ShowKey row asserts `\ShowKey` and `\ShowSolutions` reveal identically for `autoexam` (see the `\ShowKey` wiring under Changed) — if `\ShowKey` ever regresses to a no-op the row fails.
- **Per-version/solutions PDF slicing from one combined autoexam build.** A normal build of an autoexam `\versions{A,B,C,...}` document (or a `\solutions` dual/only build) now slices `<base>_A.pdf`, `<base>_B.pdf`, `<base>_A_solutions.pdf`, … out of the single combined `<base>.pdf` automatically, instead of needing a separate mode or a separate recompile per version. `autoexam.cls` tracks an absolute, never-reset page counter via `\AtBeginShipout` and writes a `<base>.vmap` sidecar (`\AutoExamVmapOpen`/`Record`/`Close`); `autoexam_run_versions` (`problem_engine.lua`) records one line per copy; the Sublime builder's `_slice_versions_from_vmap` reads the sidecar and uses `pypdf` to slice after the engine finishes.
- **Box-grid schedule renderer (`box-grid = true`): real per-cell SyncTeX inverse search for the calendar.** The `schedule` class can now draw the calendar as stacked box rows (`\schedcell`/`\schedlabel`/`\scheddivider`, driven by a new `box_grid` branch in `render_grid`) instead of an `xltabular`. It looks the same, but each cell ships during ordinary box construction, so SyncTeX records every cell against its *own* grid-file line and the existing `.schedmap` → `.synctex.gz` rewrite finally takes effect: double-clicking a calendar cell in the PDF opens the source `.tex` at the directive that produced that cell. Verified end-to-end (real `lualatex` build + `synctex edit`) in `Sublime/test_synctex_integration.py` scenarios 9–10 (distinct cells → distinct source lines) and a new `schedule__box-grid` visual scenario. Opt-in; the default renderer is unchanged. Known gap: the box grid repeats the day-name header only per page-*group*, so pair it with `month-pages = true` (header per month) for a multi-page schedule.
- **Semantic tests for four behaviors previously only substring- or pixel-checked** (a real regression could pass green before). (A) `test_metadata_resolution.py` builds a document two directories below its `coursemeta.tex` and asserts the directory walk (`course-metadata.sty:232`) finds it, `\GetCourseMetaDir` returns the `../../` prefix, and a coursemeta-relative `preamble-file` resolves from that depth; plus that a class option overrides a `coursemeta.tex` value (both a predefined and an auto-vivified key) — the precedence only asserted in comments until now. Real `pdflatex` + `pdftotext`; soft-skips without them. (B) `Bingo/test_bingo.lua` pins `bingo.lua`'s card generator: a 25-cell grid, the free space present (centred on a hand-authored card), and — for a randomized deal — 24 *distinct* non-free cells, plus determinism and the "needs ≥25 entries" guard. (C) `tests/fixtures/Notes/theorem-numbering.tex` (+ `EXPECT_TEXT`) asserts didactic's shared-counter, section-based theorem numbering renders `Theorem 1.1 … Lemma 1.3 … Theorem 2.1`, catching a per-family/flat/no-reset counter regression the pixel refs miss. (D) `Report Cards/test_report_card.lua` pins `report_card_engine.lua`'s gradebook-CSV → `\rcRenderStudent` contract: numeric current-total extraction, per-category Weight/Score/Points column mapping, blank → em dash, the `---` midrule, and the scenario cutoff cells (the weighted arithmetic itself lives upstream in the spreadsheet, so this is the engine's whole computable surface). The pure-Lua tests run in a new fast `lua-unit` job in `tests.yml` (stock `lua5.3`, no TeX toolchain — this also wires in the previously-unrun `test_shuffle.lua`); the metadata build test runs in the existing full-TeX-Live container job.

### Fixed

- **`quiz.cls`: `\ShowKey` / source `\keys` now builds the answer key.** Like the parallel `autoexam` fix, quiz raised the "Answer Key" cover label but never set `\ifsolutions`, so a compile-time key build hid the solutions + multiple-choice key (a silent no-op key). A quiz is a single copy, so `\ifkey` now simply implies `\ifsolutions`. Surfaced by the now-parametrized `test_mode_effects.py` (exam/quiz/pset in one differential file).
- **`bank.cls` catalog: multi-part problems no longer render "0a. 0b." part labels.** A catalog has no active `\question`, so the shared `\partlabel` (`\thequestion\thepartno.`) leaked `\thequestion = 0`; parts now render self-containedly as `(a) (b)`. Guarded by `test_bank_catalog.py`.
- **`Sublime/test_synctex_integration.py`: pinned `LUAINPUTS` so the test uses the checkout's engine, not a `TEXMFHOME` shadow.** It pinned `TEXINPUTS` but not `LUAINPUTS`, so LuaTeX loaded the bank engine from a stale install; the mismatch made the autoexam multi-version emit render no problem content, which *falsely* failed the stem / multi-version / document-body inverse-search scenarios. With the shadow gone those pass; the free-response and multiple-choice *solution* inverse-search remain a tracked known issue (`task_b7217810` / `task_dbeb33f6`) — the pre-collected solution box's rendered position doesn't map back to its (correct) SyncTeX record.
- **Extra-credit problems in a shuffled `{problems}` section now get the same per-problem chrome as graded problems — a full-width separator rule above and reserved answer space below.** Inside `{problems}`/`{mcproblems}`, `\extracredit` is collected and replayed *after* the permuted problems (`\pbank@xc@deferred` in `texlib-problembank.sty`), which sidesteps the engine's per-`\problem` emit loop — so a bonus floated with no divider from the last graded problem (unlike every other problem) and, in student builds, no `\workbox` answer space to write a solution in. `\extracredit` now takes an optional stretch (`\extracredit[pts][stretch]{…}`, default 1, mirroring `\problem[pts][stretch]{…}`) and emits its own trailing `\workbox{stretch}`; the deferred replay is wrapped so each bonus is preceded by the inter-problem separator rule (`\autoexam@problem@sep`), honoring the first-on-page rule through a shared `pbank_emit_sep` factored out of `pbank_problem_item` (`problem_engine.lua`). `\workbox` is `\ifsolutions`-suppressed, so an answer key stays compact but still gets its separator. Because `\extracredit` now reserves its own space, the redundant manual `\vspace{\stretch{1}}` after it in `Quizzes/quiz-template.tex` and `examples/Math181-Fall2026/quiz-01.tex` is dropped. Any existing exam that uses `\extracredit` re-paginates — the bonus now reserves answer space — intended.
- **The green `{solution}` box no longer overhangs the right text margin.** The box is sized to the full `\textwidth`, but it renders inside the exam-class `{questions}` list body whose left indent (`\@totalleftmargin` ≈ 19.5 pt) `\noindent` does not remove — so the full-width box was shifted right, insetting its left edge and overhanging the right margin by that indent. `texlib-solutions.sty` now full-bleeds it to the true margins with `\hspace*{-\@totalleftmargin}` (exactly like the inter-problem separator rule it sits beside), so both edges land on the text block. In the side-by-side multiple-choice key the box lives in a `minipage`, which zeroes `\@totalleftmargin`, so the shift is a no-op there and the narrow-column key is unchanged. (`{partsolution}` was checked for the same pattern: its prose already lands at the text margin and spans the block without a right overhang, so it is left as-is.)
- **The schedule calendar grid no longer makes LuaLaTeX log "Non-PDF special ignored!" — it is rebuilt on `tabularray`.** The grid was an `xltabular` (i.e. `longtable`) whose day cells were shaded with `colortbl`'s `\cellcolor`. Under LuaLaTeX that pairing emits one inert *empty* `\special{}` per page during `longtable`'s trial/measuring pass; the PDF backend can't emit a non-`pdf:` special and logs "Non-PDF special ignored!" on every shipout. The cell color itself is drawn through the color stack, not that special, so the warning was cosmetic — but it fired on every schedule build. `schedule.cls` now builds the grid on `tabularray`'s `longtblr`: native `\SetCell{bg=...}` shading (no `colortbl`, so no stray special), `rowhead=1` for the header that repeats atop each page (replacing `\endhead`), `hlines` plus a trailing `\hline` after the header row for its two-line separator (the vertical rules break across the gap, matching the old `\hline\noalign{\vskip2pt}\hline`), and `rowsep=0pt` so row heights — and therefore page breaks — match the old grid. `longtblr`'s cross-page "(Continued)" head/foot are suppressed so a multi-page schedule paginates as tightly as before: `contfoot`'s `empty` template drops the foot, but `conthead`'s `empty` template is a no-op in this `tabularray` release, so its text macro `\tblrcontheadname` is blanked at `\AtBeginDocument` instead. (`rowhead` also needs `[entry=none, label=none]` on the environment, or it throws a `Missing number`/`Illegal unit` dim-eval error while building its repeating-head box.) Output is visually faithful to the `xltabular` grid — header double rule, quiz/exam/holiday cell colors, month-start date boxes, per-day layout — and all seven schedule visual scenarios keep their page counts. `xltabular`, `colortbl`, and the (unused) `multirow`/`booktabs` are dropped from `schedule.cls`; the schedule visual references were regenerated.
- **Bank/autoexam engine scratch no longer litters the source folder on aux-routed command-line builds.** `problem_engine.lua` / `texlib_synctex.lua` write their build-time scratch (per-version body files, `.sco`, `.srcmap`, the SyncTeX-redirect temp) via raw Lua `io.open`, which ignores `-output-directory`. The Sublime builder papered over this by exporting `TEXLIB_AUX_DIR` so `texlib_scratch_path` could mirror the routing — but any *other* build that routed aux via `-output-directory` alone (a plain command-line verification build, another Claude Code session compiling the doc to check its own work) still dropped this scratch next to the `.tex`, on OneDrive-synced course folders no less. `texlib_scratch_path` now also honors `TEXMF_OUTPUT_DIRECTORY` (which TeX Live sets itself whenever `-output-directory` is used) as a fallback after `TEXLIB_AUX_DIR`, so those builds keep the source folder clean too. Only a build with no aux routing at all (both unset) still writes beside the source — the documented, unavoidable fallback (there is nowhere else to put it). The schedule class's own `*_schedule_grid.tex` uses a separate engine and is not covered.
- **Multi-copy / `.spl` PDF post-processing now works under Sublime, not just the CLI.** The per-version `.vmap` slicing and the `.spl` exam/solutions split are pypdf operations that ran in the builder's Python process — which under a real Ctrl+B build is Sublime's *embedded* Python (3.8, empty site-packages), where `import pypdf` fails. The steps silently no-op'd (an info line, hidden by `show_panel_on_build: errors`), so a multi-version exam produced only the combined PDF; it worked from the command-line test harness only because that runs under the system Python. The pypdf logic is now factored into `Sublime/texlib_pdfpost.py` and run in-process when pypdf is importable, else via a discovered external Python that has it (`_run_pdfpost` / `_external_python`, honoring `$TEXLIB_PYTHON`). `deploy.ps1` deploys and syntax-checks the new module.
- **`\begin{solution}` content now has working inverse search in Solutions-mode builds.** The originally-suspected root cause (`tcolorbox` defers shipout for internal measurement, like `xltabular` above) was wrong — disproved by ablation, stripping `tcolorbox` out of `{solution}` entirely still failed identically. Two separate bugs were actually stacked: (1) `problem_engine.lua` never staged a SyncTeX redirect for the solution region at all — `p.solution` was `tex.print`'d as a raw string with no file backing, so it silently inherited whatever `(file, line)` the stem's own redirect last left active; fixed with a new `pbank_stage_solution`/`emit_solution_block` pair that stages and `\@@input`s the solution's own bank-file lines, deferred through a follow-up `\directlua` token (`texlib_synctex.lua` allows only one pending redirect at a time — same pattern as `pbank_print_catalog`'s existing per-id deferred calls). (2) Separately, `tcolorbox`'s `enhanced` mode (needed for the old left-accent border) does its own internal box handling that defeats `synctex edit`'s reverse search even for correctly-tagged content, confirmed generic (not specific to solution content) and traced to `\unvbox` splicing nodes with no box-open record of its own — fixed by dropping `tcolorbox` from `{solution}` entirely in favor of plain `\colorbox`/`\parbox`, `\box` (not `\unvbox`) to insert the pre-collected content as a real nested box node, and a hand-drawn `\vrule` for the left accent. Trade-off: the box can no longer break across a page boundary the way the old `breakable` `tcolorbox` could; accepted after surveying every shipped solution (none exceed ~7 lines, none use diagrams/tables, no historical need for one). `Sublime/test_synctex_integration.py`'s Scenario 2 now passes outright — no more known-failure carve-out.
- **Schedule SyncTeX inverse search now actually works (was silently broken under `xltabular`).** The v0.2.0 claim that "clicking a calendar cell opens `template.tex` at the directive's line" held only against a hand-fabricated `.synctex.gz`, never a real build: `xltabular` (via `longtable`) defers box shipout to its output routine, by which point every cell's raw SyncTeX line has collapsed to the grid file's last line, so the `.schedmap` rewrite found nothing to remap. Two-part resolution: (1) `_rewrite_synctex_for_schedmap` now only repoints the grid-file `Input` record at the real source when at least one cell record was actually remapped — otherwise it honestly leaves clicks landing in the auto-generated grid scratch file instead of confidently pointing at a wrong source line; (2) the new opt-in `box-grid` renderer (above) makes per-cell attribution genuinely correct. Under the default `xltabular` renderer, inverse search remains the honest grid-file fallback.

### Removed

- **The "All Versions" / "All Versions (Solutions)" Sublime build modes, and the standalone `build_versions.py` parallel multi-version builder.** Both are superseded by the automatic per-version/solutions PDF slicing above — a normal default build now does what these did, faster (one compile instead of one recompile per version, even parallelized across cores) and without extra clicks. ### Limitations (schedule SyncTeX)

- **Module templates renamed to unique jobnames** (`autoexam-template.tex`, `schedule-template.tex`, …). Every module previously shared the jobname `template`, so under a recursive `TEXINPUTS` a build could resolve `template.tex` to a *different* module's file (e.g. an exam build pulling in the schedule template, erroring on `lecture-days`). Unique names eliminate the collision for every build method; `smoke_test.py` and the builders adapt automatically (the jobname derives from the filename).
- **Class-option values with commas no longer emit a spurious "unused global option" warning.** The kernel's option parser splits a braced value like `{May 2, 2026}` at its inner comma into a stray fragment; each class now discards those leftovers (the real value is read brace-aware from `\@raw@classoptionslist`).

### Tooling

- **`build_versions.py` is documented in the top-level README** (it previously appeared only in `Sublime/README.md`).

### Added

- **Visual scenario packs (`smoke_test.py --scenarios`).** A tiered visual-test
  layer on top of the per-module suite: each scenario under
  `tests/scenarios/<area>/<name>/` is a self-contained template (metadata inline
  via `\metasetup`) exercising one configuration, built and pixel-diffed against
  `tests/visual_refs/<area>__<name>-*.png`. Tiers via an optional `tags` file
  (`core` runs by default, `full` only with `--full`); `--scenarios [AREA...]`
  filters by area, `--update-refs` regenerates. Ships packs for **Schedule**
  (`landscape-mwf`, `portrait`, `month-pages`, `summer-intensive`,
  `mid-week-start`, `recitations`, `no-quiz`), **Report Cards** (`standard`,
  `multi-student`), and **Syllabi** (`standard`, `long`) — covering orientation,
  month-pages, a 5-day intensive, partial weeks, recitation columns, a quiz-free
  grid, multi-student report cards, and multi-page syllabi. Local-only, like all
  visual checks (references are environment-specific).
- **`smoke_test.py` now verifies rendered content, not just build success.**
  After each successful build it (1) extracts the PDF text with `pdftotext` and
  asserts per-module expected substrings are present (`EXPECT_TEXT`), and (2)
  checks that key generated artifacts are non-empty (`EXPECT_ARTIFACT_NONEMPTY`
  — e.g. the schedule's `*_schedule_grid.tex`). This catches the "compiles green
  but renders blank/garbled" class a build-only check misses — the empty
  schedule grid is now a hard failure. Opt-in `--visual` pixel-diffs each page
  of the deterministic modules (`VISUAL_MODULES`: Schedule, Report Cards,
  Syllabi, Notes) against committed references in `tests/visual_refs/`
  (`--update-refs` to regenerate), catching layout regressions like the
  row-stub that text checks can't see. New flags: `--no-content`, `--visual`,
  `--update-refs`, `--dump-text`. Every external-tool check (poppler /
  ImageMagick) soft-skips when its tool is absent, so a bare TeX install still
  runs build-only. CI (`smoke.yml`) installs `poppler-utils` so content checks
  run on every push; visual regression stays a local aid (its references are
  rendering-environment-specific). Builds now run multiple passes — re-running
  while the `.log` requests it (latexmk-style) — so `\pageref{LastPage}` and
  other cross-references resolve in the rendered output ("1 of 2" rather than
  the one-shot "1 of ??").
- **`recitation-days` meta key on the `schedule` class.** Now registered in the
  `meta` family (mirroring `quiz-days`: a `\clist_gset` store + an expandable
  `\GetRecitationDays`), so `\meta{recitation-days = R}` adds a capacity-0
  recitation column. Previously `\GetRecitationDays` was an empty
  `\providecommand` with no key to set it, despite the README documenting one —
  `\meta{recitation-days=...}` errored with "unknown key".
- **`month-pages` meta key on the `schedule` class.** `\meta{month-pages = true}` typesets each calendar month as its own table on its own page (`\newpage` between months) instead of one continuous term-long table. A week straddling a month boundary is repeated — at the foot of the earlier month and the head of the next — so every month shows complete weeks, wall-calendar style; week numbers stay continuous across the term. Default (`false`/omitted) is unchanged. Implemented as a page-group partition in `render_grid` (one group = whole term by default, one group per month when enabled), with the table header/lastfoot boilerplate re-emitted per group.
- **Per-call variable overrides on `\problem`.** Inside `\begin{problems}…\end{problems}`, `\problem{id}` accepts a trailing optional argument that pins random variables to specific values for that one instance: `\problem{quadratic}[a=1, b=2, c=3]` solves `x²+2x+3` instead of a freshly-sampled quadratic. Partial fixes work too — `\problem{quadratic}[a=1]` leaves `b` and `c` random. The engine adds a `fixed[]` table that `set_var`/`set_rng`/`calc_var`/`pick_*` consult before writing, so a bank entry's own randomisation calls become no-ops on locked names. `push_scope`/`pop_scope` save and restore the table around each problem, so the override is local. `\importproblem` was upgraded to use the same lock semantics.
- **`texlib-problembank.sty`** — single source of truth for the shared problem-bank LaTeX glue: engine loader, `\setvar`/`\setrng`/`\calcvar`/`\get`, the four `\pick*` commands, `\getlist`/`\geti`/`\foreachpick`, the `\begin{problem}` environment, `\getproblem` (+ `\useproblem`/`\reqproblem`), `\ppart`, `\@problem@item`, `\loadbank`, `\importproblem`, and `\providecommand` defaults for `\workbox`/`\autoexam@problem@sep`. Required by `autoexam.cls` and `quiz.cls`; collapses ~120 lines of duplicated code per class to a single `\RequirePackage`.
- **`texlib-corepkg.sty`** — universal package bundle: fontenc/lmodern/geometry, expl3/xparse/ifthen/etoolbox, xcolor, the amsmath family, `texlib-mathutils`, graphicx, tikz/pgfplots + standard libraries, and hyperref with the canonical `\hypersetup` used across every TeXLib class. `basic-utilities.sty` now `\RequirePackage`s this and adds its own extras (siunitx, caption, tasks). `autoexam.cls`/`quiz.cls` load it directly; `didactic.cls`/`pset.cls`/`report-card.cls` inherit it transitively via basic-utilities. Each of the heavier classes shrank by 8–11 lines.
- **`texlib-itemfmt.sty`** — `\encircle` (the circled-token decoration) and `\extracredit` (extra-credit `\question` variant), previously duplicated across `autoexam.cls` and `quiz.cls`. `\extracredit` is now available to autoexam too.
- **`texlib-pagestyle.sty`** — `\SetTeXLibExamRules` bundles `\pagestyle{headandfoot}` + `\firstpagefootrule` + `\runningheadrule` + `\runningfootrule`. Sibling to `texlib-footer.sty` (which serves the fancyhdr-based classes the same way).
- **`texlib-thmenv.sty`** — the union of `\newtheorem*` declarations from `autoexam.cls` (3) and `quiz.cls` (14): `thm*`, `defn*`, `cor*`, `prop*`, `lem*`, `conj*`, `ques*`, `prob*`, `exam*`, `ex*` (legacy short alias), `notn*`, plus the remark-style `rmk*`/`recall*`/`case*`/`desiderata*`. Both classes inherit the full set.
- **Shared `\the*` aliases in `course-metadata.sty`.** `\theCourseSubject`, `\theCourseNumber`, `\theCourseTitle`, `\theCourseSection`, `\theCourseRoom`, `\theCourse`, `\theInstructor`, `\theInstitution`, `\theSchool`, `\theSeason`, `\theYear`, `\theTerm` are all `\providecommand`s here now, so no class needs to declare them and there's no risk of a `\newcommand` clash if a future metadata key reuses one of those names. Class-specific aliases (`\theExamNumber`, `\theQuizNumber`, …) stay in their respective classes.
- **`tests/fixtures/Exams/fix-test.tex` + `fix-bank.tex`** — exercises the new `\problem[fix]` syntax. Registered as a smoke-test entry so the feature is covered on every CI run.
- **`smoke_test.py` now collects `.cls` files from module subdirectories** when assembling each build's temp dir, so test entries under `tests/fixtures/<Module>/` can use a sibling module's class.

### Changed

- **Test assets consolidated under `tests/`.** The top-level `Test/` directory is gone — its feature fixtures moved to `tests/fixtures/Exams/` and `tests/fixtures/Metadata/` (joining `tests/scenarios/` and `tests/visual_refs/`), removing the confusing `Test/` vs `tests/` case clash and the empty per-module scaffolding dirs. `MODULES`/`EXPECT_TEXT` in `smoke_test.py` were repointed accordingly; the fixtures still build on every push.
- **`examples/Math181-Fall2026` expanded into a full course folder and wired into CI.** Added `syllabus.tex`, `schedule.tex`, and a multi-version `exam-01.tex` + `bank.tex` (`\versions{A,B,C}` + `\shufflepages`) alongside the existing lecture and quiz — all five documents share one `coursemeta.tex` and are now registered as build-only smoke fixtures, so the documented end-to-end workflow can no longer silently rot when a class changes. `smoke_test.py` gained a `CLASS_HOME_MODULE` map (+ `detect_class`): a build whose source lives outside its document class's home module (e.g. an `examples/` `{quiz}`/`{schedule}` doc) now gets that module's default includes (`quiz-instructions.tex`, the schedule `.lua` helpers) copied into the build dir — the comma-path-safe way, since the recursive `TEXINPUTS` fallback can't reach them under a comma-bearing root.
- **The `schedule` environment no longer emits the title banner itself.** `\maketitle`/`\scheduletitle` must now be called explicitly in the document body (canonically right after `\begin{document}`, before `\begin{schedule}`), matching every other TeXLib class. This lets a schedule run title-less (e.g. when the banner lives elsewhere) by simply omitting the call. Existing schedule documents need one added `\maketitle` line; the canonical `Schedule/template.tex` and the README tutorial were updated accordingly.
- **`autoexam_engine.lua` → `problem_engine.lua`.** The file is shared between `autoexam` and `quiz`; the old name implied otherwise.
- **Lua engine function names: the shared ones gained a `pbank_*` prefix.** `pbank_problem_item`, `pbank_apply_fix`, `pbank_set_bankfile`, `pbank_inject_part`, `pbank_first_on_page`, `pbank_part_*`, `pbank_stretch_list`, `pbank_pending_*`, and the new `pbank_suppress_redirect` flag. Autoexam-specific functions kept their `autoexam_*` prefix (`autoexam_run_versions`, `autoexam_versions`, `autoexam_shuffle_pages`, `autoexam_write_srcmap`, `autoexam_read_body`, `autoexam_scorepage`, `autoexam_gradingrow`) — they would only ever be called by autoexam.cls.
- **`\loadbank` is now defined once in `texlib-problembank.sty`.** It activates the SyncTeX bank-file redirect via `pbank_set_bankfile()` so inverse search from the PDF lands in the bank source file (previously: only quizzes called this; autoexam ran with the redirect dormant to avoid a multi-version input-stack overflow).
- **The bank SyncTeX redirect now writes ONE reused scratch file per job** (`<jobname>_synctex.tex`) instead of one `<jobname>_synctex_<id>.tex` per problem. SyncTeX records the bank path, never the scratch file, and redirects are served strictly sequentially (each `\@@input` read to EOF and closed before the next is staged), so a single overwritten file is safe and course folders no longer accumulate one orphan `.tex` per problem.
- **Inline-defined problems now get inverse search into the document.** A `\begin{problem}{id}...\end{problem}` written directly in an exam/quiz with no `\loadbank` is attributed (via `status.filename`) to the file it lives in, so double-clicking it in the PDF jumps to its `\begin{problem}` block in that source — line-accurate, through the same single reused scratch file, no per-problem `_prob_` temp. Works in single- and multi-version builds for problems defined in the preamble or (single-version) body; problems in a multi-version body-replay file still use the per-problem fallback. `\loadbank` now restores the previous bank context after `\input` (stack-based, nesting-safe) so a `\begin{problem}` placed *after* a `\loadbank` is attributed to the document, not the stale bank file.
- **Quiz default instructions now support SyncTeX inverse search.** The boxed default wording moved out of `quiz.cls` into a real `Quizzes/quiz-instructions.tex` that the class `\input`s (boxed) when a quiz sets neither `quiz-instructions-file` nor an inline `quiz-instructions` — so double-clicking the instructions in the PDF jumps to that source, mirroring autoexam's `autoexam-instructions.tex`. Edit `quiz-instructions.tex` to change the default for every quiz. (Per-quiz overrides are unchanged: `quiz-instructions-file=` is unboxed/full-control; `quiz-instructions={…}` is inline.)
- **`quiz-instructions-file` can now be set course-wide in `coursemeta.tex`.** It's now a course-metadata key (defined in `course-metadata.sty` alongside `bank-path`), so `\metasetup{ quiz-instructions-file = my-quiz-instructions }` in `coursemeta.tex` points every quiz in the course at one local instructions file — resolved relative to the coursemeta directory (`\GetCourseMetaDir`), so it's found from any quiz folder regardless of depth. Resolution mirrors `bank-path`: tried as-is first (per-quiz, document-relative), then coursemeta-relative. Still overridable per quiz via `\meta{ quiz-instructions-file = … }`, and unset → the shared default `quiz-instructions.tex`. Defining the key in `course-metadata.sty` rather than `quiz.cls` is deliberate: `coursemeta.tex` is read before the class body, so a quiz-only key would be missed (and would clash with the metadata engine's unknown-key handler). Exams and other classes that load `coursemeta.tex` simply ignore the key. Covered by a new smoke-test scenario (`tests/scenarios/quiz/coursemeta-instructions/`).
- **Smoke-test scenarios can now make text assertions (`expect-text`).** A scenario that ships an `expect-text` file (PDF substrings, one per line) asserts rendered content directly and — when it carries no reference PNGs — skips the pixel diff, so a feature about *which text renders* (not layout) needs no committed image ref. Scenarios may also ship their own `coursemeta.tex` to exercise course-metadata keys. First user: the `quiz/coursemeta-instructions` pack.
- **Builds now leave a single inverse-search file: `<base>.synctex`.** The Sublime builder's post-process (`_finalize_synctex`) decompresses lualatex's `<base>.synctex.gz` to a plain `<base>.synctex` and deletes the `.gz`; because a viewer reads an uncompressed `.synctex` directly, SumatraPDF no longer spawns its `<base>.synctex.gz.sum.synctex` decompression cache. The build-time SyncTeX scratch `<base>_synctex.tex` (and any legacy per-problem `<base>_synctex_<id>.tex`) is removed too. Net: one hidden `.synctex` per document instead of up to three files. The schedule `.schedmap` → SyncTeX rewrite is unaffected — it runs on the `.gz` before decompression. (Raw command-line builds outside the Sublime builder still produce the `.gz` + scratch; only the builder finalizes.)
- **Multi-version exams now use the bank redirect too.** `autoexam_run_versions` no longer sets `pbank_suppress_redirect=true`. The flag existed to dodge a LuaTeX input-stack overflow when the version loop re-inputs every problem, but the `texlib_synctex.lua` helper closes each `\@@input` at EOF so the inputs pop and stack depth never accumulates (verified: a 3-version exam — 36 bank inputs — builds cleanly even with `max_in_open` forced to 20). Result: multi-version inverse search lands in the bank (not per-problem `_prob_` temp files), and these builds also produce just the single reused scratch file. The flag is retained as an explicit per-call override.

### Removed

- **`\newproblem` and `\dupproblem` are gone — define problems with the `\begin{problem}` environment.** The environment does everything the commands did (define in a bank file, the preamble, or the body) plus supports SyncTeX inverse search, so there's now one canonical way to author a problem. No live document used the commands. Migration is mechanical: `\newproblem{id}{meta}{body}[ans]` → `\begin{problem}{id}[meta]` *body* `\solution` *ans* `\end{problem}`. The dead `define_problem()` Lua backend was removed too (`define_problem_from_env()` remains).

### Fixed

- **`texlib-problembank` no longer forces every loading document onto LuaLaTeX.**
  Its load-time engine loader ran `\directlua` unconditionally, so any
  non-LuaTeX (pdflatex) class that `\RequirePackage`s it — e.g. `didactic`,
  which auto-loads it so lecture notes can optionally `\getproblem` — died at
  load with "Undefined control sequence `\directlua`". The loader is now guarded
  by `\ifdefined\directlua`: under LuaLaTeX it loads as before; under any other
  engine the load is skipped and the bank macros route through an internal
  `\pbank@lua` bridge that raises a clear "Problem-bank features require
  LuaLaTeX" error — but only if a bank command is actually used, so notes/handout
  documents that merely load the package still compile under pdflatex. The
  `\directlua` primitive itself is left untouched (not shadowed), so other
  packages' `\ifdefined\directlua` engine probes still work.
- **Schedule grid no longer leaves a vertical-rule stub below the last row.** The table's bottom edge used to show a short fragment of the `WEEK`/day column rules hanging beneath the final week — longtable was synthesising a phantom trailing row from the non-empty `\endlastfoot` (`\noalign{\vskip -12pt}\hline`). `render_grid` now ends every row, including the last, with `\tabularnewline \hline` and uses an empty `\endlastfoot`, so the last row's own rule is the clean bottom edge. (The same fix applies to each month's table in `month-pages` mode.)
- **SyncTeX bank-file redirect no longer fires "missing \item" errors for problems past line ~1000.** The redirect path pads its temp file with blank lines so the served content lines up with the bank's true source lines (for accurate inverse-search attribution). Each blank line was tokenising — under the default `\endlinechar=13` — to a `\par` token; exam.cls's `\trivlist` starts complaining once roughly 1000 `\par`s have stacked inside a `\question` item, breaking single-version exams and quizzes whenever a problem lived past line ~1000 of the bank. `typeset_problem` now brackets the padding region with `\endlinechar=-1\relax` on line 1 and `\endlinechar=13\relax` on `sm.line-1`, so the blank lines emit zero tokens (no `\par`) and only one harmless `\par` fires right before the real content begins. SyncTeX line attribution is preserved — clicks still land on the correct bank line.
- **`\importproblem` overrides are now actually locked.** Previously its overrides were set via direct `set_var` calls, which the imported file's own `\setrng` could clobber. It now routes through `pbank_apply_fix` and gets the same `fixed[]`-table semantics as `\problem[a=1,b=2]`.

### Cleanup

- Stripped duplicate `\RequirePackage` lines (`fontenc`, `lmodern`, `geometry`, `xparse`, `expl3`, `etoolbox`, the amsmath family, `graphicx`, `tikz`/`pgfplots`, `hyperref` + `\hypersetup`) from `didactic.cls`, `pset.cls`, and `report-card.cls` — all now come through `basic-utilities` → `texlib-corepkg`.
- Removed stray build artifacts (`Notes/template.aux`/`.log`/`.out`/`.toc`, `Problem Sets/template.pdf`/`.synctex.gz`) that had escaped `.gitignore`.

## [0.1.1] — 2026-05-23

Patch release: aux-directory routing for the Sublime builder + theorem-environment polish across the document classes.

### Added

- **Starred (unnumbered) theorem environments in `didactic` and `pset`.** Every theorem-family environment gets a `*` twin (`theorem*`, `lemma*`, `definition*`, `proposition*`, `corollary*`, etc.) plus, in didactic, the short-alias starred forms (`defn*`, `conv*`, `ques*`, `exer*`, `exam*`, `notn*`, `chal*`). Same tcolorbox styling as the numbered versions; consume no counter. Use for one-off results you don't want to clutter the numbering with.
- **Context-sensitive top-level `enumerate` labels in `didactic`.** Inside bold-titled theorem/definition environments, top-level enumerate items render as bold roman (`i.`, `ii.`, `iii.`); everywhere else (body text, remark/question/recall boxes), they render as italic roman. Implemented via `\AtBeginEnvironment` raising a boolean that's read at label-typesetting time, so a bare `\begin{enumerate}` adapts to its surroundings with no per-list configuration.
- **Bold roman top-level `enumerate` labels in `pset` and `autoexam`.** Matches didactic's bold-roman-in-theorem-environment look, applied unconditionally since neither class has a mixed-context need. `\ref` to enumerate items prints `i`, `ii`, ... consistent with the visible label.
- **`didactic` auto-sets the section counter to `\GetUnitNumber` in `\maketitle`.** Subsections now render as `N.1`, `N.2`, ...  and section-numbered theorems pick up the unit prefix without the per-file `\setcounter{section}{...}` boilerplate. Guarded against documents that omit the unit-number metadata key.

### Fixed

- **`Sublime/texlib_builder.py` now honors the LaTeXTools `aux_directory` setting** (the template ships with `"<<temp>>"`, which was previously ignored). The builder routes the engine via `-output-directory` to a stable per-document temp dir under `%TEMP%\texlib-aux\<hash>\`, then copies the PDF, `.synctex.gz`, and any `.spl` signal back next to the source. Net effect: `.aux/.log/.out/.toc/.bcf/.bbl/.fls/.fdb_latexmk` stop accumulating in source directories and OneDrive doesn't see them as changes on every Ctrl+B. biber invocations now use `--input-directory` / `--output-directory` so biblatex cross-references still resolve when aux routing is active. Set `aux_directory` to `""` or `"<<root>>"` in `LaTeXTools.sublime-settings` to opt out and restore the old in-source behavior.
